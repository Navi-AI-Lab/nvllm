# Path B Z1 — controlled cache-pin causality test summary

## Verdict

**Compile-artifact identity is the dominant cause of the FULL+β-coop replay-coherence FAIL.** The discriminator found in X-trial log analysis (62 MB vs 73 MB inductor-compiled artifact) does NOT merely correlate; it CAUSES the verdict in 9 of 10 trials.

Per the user's verdict framework: this lands at **"5/5 PASS + cache_reused on all 5 → causality basically closed"** for the good-cache experiment, with the bad-cache control showing strong (but not 100%) FAIL bias.

## Two cache artifacts (both produced by inductor at the same cache key)

Same torch AOT compile cache key (`9a5549f23a178e35a9a3e9b4bed7adf1d137d22f3fc06ef8048d589e5d625721`) was generated by both compiles, but with distinct sha256s. This is inductor non-determinism: same input, different output binary.

| Label | Disk size | Log-reported size | sha256 |
|---|---|---|---|
| GOOD | 73,413,669 B (~70 MB) | 62,142,045 B (~62.1 MB) | `651e00bd5997bacd9a062da66e6c9a078ed3c4469c27c715d8b025041a2a8264` |
| BAD | 84,885,730 B (~81 MB) | 73,614,088 B (~73.6 MB) | `af68c498c6ee45b60165d584a870f2f072068153a7c76d9592fc0097efe63c80` |

The "log-reported" size is what `torch.compile` prints as "collected artifacts: N entries, M artifacts, X bytes total" — the X-trial discriminator.

## Implications for Z

**Production fix shape: persist a known-good torch.compile AOT cache across container starts.**

Concrete next steps (NOT executed in this experiment, deferred to a separate Z-design):

1. **Lock the AOT cache via host-mount in the production serve scripts** (`scripts/serve-cute*.sh`), not just the spike `c2_full_layer_bisect.sh`. Mount a host-persistent dir to `/root/.cache/vllm/torch_compile_cache/torch_aot_compile/`.

2. **Bootstrap a known-good cache once** — could be a one-shot script that boots the model with audit-OFF, runs c2_replay_coherence, accepts only if `unique=1`, and saves that cache as the canonical artifact. Or seed from a verified-good build artifact.

3. **Verify against more model configs.** This experiment used Qwen3.5-27B-NVFP4 with lower-8 β-coop on FULL_AND_PIECEWISE. The same cache may not be valid for other layer subsets, models, or cudagraph modes — each combination has its own cache key.

4. **Report upstream.** torch.compile / inductor producing two distinct binaries for the same cache key is the underlying issue. This is upstream territory; our locking is a workaround.

## What this experiment did NOT prove

- **The bad.5 PASS** says there's a smaller (~20%) source of non-determinism that even a locked-bad artifact can occasionally evade. We have NOT identified what that is.
- **The X.1-X.5 artifact sha256s are NOT preserved** — those containers were torn down before the experiment. We can only correlate by log-reported size, not by binary identity. The Z1 artifacts (good and bad) are preserved on disk for further analysis.
- **The 70 MB / 81 MB disk-vs-log size discrepancy** isn't fully explained. Disk size includes more than the "collected artifacts" measure; could include compile metadata, modelinfos, etc. The discriminator that matters is the log-reported X bytes.

## Status

- v2 closeout's "stochastic FAIL inside FULL+β-coop replay/kernel behavior" is **superseded** by this finding. The FAIL is dominated by which inductor artifact got compiled.
- Path B is closed: dispatch, Python decisions, metadata layout, and static allocation hypotheses are ruled out (X-trial + Y/Y2 + Z1 evidence chain). Bug is upstream torch.compile non-determinism + a smaller residual source.
- v3 (mlp_partial_fp32 host-reset) is **NOT the right fix.** The right fix is artifact pinning.
