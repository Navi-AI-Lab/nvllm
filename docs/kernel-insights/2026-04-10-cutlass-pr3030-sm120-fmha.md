## What Was Borrowed

This PR is the primary reference for SM120/SM121 CuTe DSL attention design
decisions. We did not copy code directly — instead, we used its findings to
inform our kernel architecture.

### Design Decisions Informed by PR #3030

| Decision | PR #3030 Finding | Our implementation |
|---|---|---|
| CpAsync over TMA for KV loads | CpAsync matches or beats TMA across 31 configs; warp specialization overhead doesn't amortize with narrow m16n8k MMA | `kernel.py`: CpAsync for all K/V page loads |
| FP8 m16n8k32 for QK pass | SM120-native instruction, 2x throughput vs BF16 m16n8k16 | `kernel.py:qk_pass` uses FP8 MMA |
| BF16 m16n8k16 for PV pass | Standard SM80-compatible MMA; V dequanted FP8→BF16 inline | `kernel.py:pv_pass` uses BF16 MMA |
| exp2-based online softmax | Flash Attention v2 pattern: `exp2(x * log2(e) * scale - max)` | `kernel.py:online_softmax` |
| No warp specialization | SM120's narrow MMA keeps all threads busy; dedicated DMA warp is overhead | All threads participate in load + compute |
| Swizzle(B, 4, 3) for TMA | Wrong swizzle causes `CUDA_ERROR_ILLEGAL_INSTRUCTION` on SM120 | Spec Section 4.3 |
| 101 KB SMEM budget | SM120: `sm120_smem_capacity_bytes = 101376` — less than half SM90 | Spec Section 4.9 tile config verification |
| No tcgen05 / No TMEM | SM120 uses `mma.sync.aligned` (SM80-era), not datacenter Blackwell | Accumulators in registers, not TMEM |

### Source Files Referenced

All in `examples/python/CuTeDSL/blackwell_geforce/` at PR #3030 head:

- `flash_attention_v2.py` — BF16 CpAsync and TMA variants
- `fp8_flash_attention.py` — FP8-optimized kernel
- `benchmark_fp8_vs_bf16.py` — Performance comparison suite

**Why:** SM120 (Blackwell GeForce / GB10) has a fundamentally different MMA
architecture than datacenter Blackwell (SM100, SM103). PR #3030 is the only
public reference for building CuTe DSL attention on SM120, providing both
working code and performance data that prove CpAsync > TMA and quantify the
FP8 throughput advantage.

**How it was adapted:**
- Our kernel targets paged KV cache (PR #3030 uses contiguous KV)
- GQA head group iteration added (PR #3030 tests MHA only)
- Split-KV reduction for long sequences (not in PR #3030)
- vLLM AttentionBackend integration (not in PR #3030)
- Disk cache from b12x (separate source, see companion doc)

## Performance Context from PR #3030

On DGX Spark (SM121a):
- **FP8 kernel:** 35.3-42.4 TFLOPS depending on head_dim
- **BF16 CpAsync:** 50-85 TFLOPS across typical configs
- **FP8 vs BF16 speedup:** up to 1.38x at larger batch sizes
- **CpAsync vs TMA:** geometric mean 1.07x in favor of CpAsync

These numbers are for standalone attention kernels, not end-to-end serving.
Our end-to-end profiling (Task 1) showed attention is 4-6% of total GPU time,
so kernel-level gains translate to modest serving improvements. The project
value is in control, not raw performance.
