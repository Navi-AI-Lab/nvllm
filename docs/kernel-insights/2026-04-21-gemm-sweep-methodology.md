## Decisions

1. **Microbench shortlists, E2E validates.** Microbench captures ~20 legal
   configs per shape × 10 M values in a few minutes — fast iteration. It's
   static-kernel-shape timing, so scheduling/pipeline effects from CUDA
   graphs and real request streams are invisible. Use it to cut the search
   space to ~12 configs; let live E2E traces validate no silent regression.

2. **Shortlist on min μs, not mean.** Tile choice is compile-time-static; the
   noise floor is launch/measurement jitter and doesn't bias which tile is
   best. Min captures "best case reliably achieved" without being fooled by
   a few outlier slow iterations.

3. **Runtime env-var dispatcher** instead of N rebuilds. Memory
   `feedback_delegate_builds` says rebuilds are tmux-delegated; cutting 50 min
   per candidate × 11 extra configs = ~9 hours saved. The `Tuneable` template
   has the same instantiation surface as the production templates — the
   `smoke_M256` self-check in `summary.md` confirms no template-indirection
   overhead (Δ < 0.11% vs direct production path at decode M).

4. **GSM8K as correctness gate** per memory `feedback_post_quant_sanity`. Any
   config that fails 8/8 is dropped before entering the verdict table. All 12
   passed 8/8 on 2026-04-21 — no corrections needed this run.

5. **Hybrid architecture accounting.** 48/64 Qwen3.5-27B layers are
   `linear_attention` (SSM, BF16, in the NVFP4 `ignore` list); only 16 are
   NVFP4 `full_attention`. All 64 layers carry MLP pairs. Aggregate call
   mix: qkv_proj 16, o_proj 16, gate_up_proj 64, down_proj 64 per forward.
   MLP GEMMs dominate 4:1, so `gate_up_proj` / `down_proj` shape tuning
   weight matters more for aggregate decode throughput.

6. **Decode-dominated E2E interpretation.** The B.2.3 trace workload (60
   iterations @ concurrency 4, max_tokens=256, fixed prompt) is decode-heavy;
   Stream-K M≤16 path carries ~99% of NVFP4 GEMM calls. The mid-batch
   (M=17–256) band where the non-smoke configs differ is rare. Consequence:
   E2E deltas between configs sit within ±1%. **The microbench is the
   authoritative signal for the mid-batch dispatch choice**; the E2E traces
   serve as the no-silent-regression gate, not as the ranking authority.

7. **NVTX M-tag best-effort.** The plan called for a per-kernel
   `fp4_gemm_bf16 M=<m>` NVTX annotation; the `__has_include(<nvtx3>)`
   guard fell back to no-op at compile time. Punted per the spec — shape
   alone distinguishes the GEMMs, and CUTLASS emits distinct mangled
   symbols per instantiation anyway (see Phase A summary).
